# WebRTC

## Overview

WebRTC handles two distinct use cases:

1. **Live stream** — monitor sends video+audio tracks to viewer in real time
2. **Data channel** — viewer requests segment metadata, coverage maps, and segment blobs from monitor on demand

Both use the same `RTCPeerConnection` infrastructure. The mode (`'live'` or `'data'`) is declared in the initial `offer-request` signal. A data-only connection skips adding media tracks entirely, avoiding unnecessary bandwidth.

## Files

| File | Role |
|------|------|
| `peer.ts` | `RTCPeerConnection` factory, ICE helpers |
| `signaling.ts` | NIP-59 gift-wrap send/receive, signal TTL + dedup |
| `signal-router.ts` | Single global kind-1059 subscription routing to both sides |
| `viewer-peer.ts` | Viewer session management, Promise-based request API |
| `monitor-peer.ts` | Monitor session management, data channel message server |

## Peer Setup (`peer.ts`)

All connections use a single STUN server:
```
stun:stun.l.google.com:19302
```
No TURN server is configured. Cross-network connections behind symmetric NAT may fail. If NAT traversal is required, add a TURN entry to `ICE_SERVERS`.

### Non-trickle ICE

`waitForIceGathering(pc, timeoutMs=5000)` delays sending SDP until `pc.iceGatheringState === 'complete'`. All ICE candidates are embedded in the SDP body. This reduces Nostr relay events per handshake from ~15 (trickle: 1 offer + 1 answer + N candidates each side) to 3 (offer-request + offer + answer).

The 5s timeout guards against STUN servers being unreachable — the connection proceeds with whatever candidates were gathered.

## Signaling (`signaling.ts`)

### Signal Message Format

```typescript
interface SignalMessage {
  type: 'offer-request' | 'offer' | 'answer' | 'hangup' | 'ping' | 'pong';
  sessionId: string;      // UUID generated by viewer per connection attempt
  sdp?: string;           // offer / answer SDP
  mode?: 'live' | 'data'; // offer-request only
  sourceId?: string;      // live mode: specific source stream (legacy; prefer channelId)
  channelId?: string;     // live mode: named channel (video+audio composite)
}
```

### NIP-59 Gift-wrap

Signals are sent as kind 1059 events using `giftWrap(inner, privkey, recipientPubkey)`. The outer event uses a random ephemeral key and randomized `created_at`, hiding both sender identity and timing from relay operators.

On receive: `giftUnwrap(event, privkey)` decrypts to get the inner rumor. The inner rumor's `created_at` is checked against a 60-second TTL — stale signals are dropped.

### Deduplication

`seenEventIds` is a module-level `Set<string>` cleared every 60s. Because relays may replay events on reconnect, and because the randomized `created_at` means `since` filters don't work, dedup by outer event ID is the only reliable mechanism.

## Viewer Side (`viewer-peer.ts`)

### Session Management

Sessions are keyed by `monitorPubkey`. One session = one `RTCPeerConnection` + optional `RTCDataChannel`.

```typescript
sessions: Map<monitorPubkey, {
  pc: RTCPeerConnection;
  sessionId: string;
  dataChannel: RTCDataChannel | null;
  mode: 'live' | 'data';
}>
```

`connectPromises` prevents concurrent duplicate connection attempts to the same monitor.

### Connection Flow

```
ensureConnection(privkey, myPubkey, monitorPubkey, mode)
  │
  ├─ check sessions.has(monitorPubkey) → already connected? return
  ├─ check connectPromises.has(monitorPubkey) → in progress? return same Promise
  │
  └─ new Promise:
       sendOfferRequest(...)          → monitor triggers handleOfferRequest
       wait for pc.ondatachannel      → monitor's data channel arrives
       channel.onopen                 → resolve
       30s timeout → fail() + closeSession
       ICE 'failed'/'closed' → fail()
       ICE 'disconnected' → 8s grace timer → fail()
```

### Session ID Adoption

If a prior connection attempt timed out, it may leave a stale session in `sessions` with the old `sessionId`. When a new offer arrives (always direct response to an offer-request), `handleViewerSignal` adopts the new `sessionId` if the session has no open data channel yet:

```typescript
if (session.sessionId !== msg.sessionId && msg.type === 'offer' && !session.dataChannel) {
  session.sessionId = msg.sessionId; // adopt incoming session ID
}
```

### Request API

All viewer-side requests follow a Promise pattern. Each in-flight request is tracked in a map keyed by a correlation field (e.g. `requestTime` or `segmentId`). The data channel message handler resolves or rejects the right promise.

```typescript
// Example: request coverage map
requestCoverageMap(privkey, myPubkey, monitorPubkey, mimePrefix?)
  → ensureConnection(...)
  → dc.send({ type: 'coverage-request', mimePrefix })
  → Promise resolved when 'coverage-map' response arrives

// Example: request segment
requestSegment(time, privkey, myPubkey, monitorPubkey, mimePrefix?)
  → ensureConnection(...)
  → dc.send({ type: 'segment-request', time, mimePrefix })
  → accumulate chunks until index === total - 1
  → reassemble → resolve with { blob, mimeType, startTime, endTime, ... }
```

### Boundary Navigation

```typescript
requestSegmentsAfter(after, count, ...) → [SegmentMeta, ...]
requestSegmentsBefore(before, count, ...) → [SegmentMeta, ...]
```

These return metadata only (no blobs). Use `requestSegmentById` to fetch blobs for specific IDs.

## Monitor Side (`monitor-peer.ts`)

### Session Management

```typescript
sessions: Map<viewerPubkey, MonitorSession>
```

Each session has an idle timeout (default 120s). Any data channel message resets the timer. Expiry sends a `hangup` signal and closes the connection.

### Connection Flow

```
handleOfferRequest(privkey, monitorPubkey, msg, fromViewerPubkey)
  │
  ├─ close any existing session for this viewer
  ├─ createPeer()
  ├─ if mode === 'live': add tracks from activeStreams
  │    channelId → channelActiveSources override or ChannelConfig default
  │    sourceId  → legacy single-source path
  │    neither   → composite all open sources
  ├─ createDataChannel('data')
  ├─ createOffer → setLocalDescription
  ├─ waitForIceGathering()
  └─ sendOffer(...)
```

### Live Stream Track Selection

Priority order for building the composite `MediaStream`:
1. `channelActiveSources.get(channelId)` — sources from currently active `RecordAction` (pushed by `SentrySection._updateChannelActiveSources()`)
2. `ChannelConfig.videoSourceId` / `audioSourceId` — channel default/fallback
3. All open sources — if no channel specified

### Data Channel Message Protocol

All messages are JSON strings. The monitor handles:

| Request type | Response type(s) | Description |
|---|---|---|
| `source-list-request` | `source-list` | List of open sourceIds |
| `coverage-request` | `coverage-map` | `[[startTime, endTime], ...]` ranges |
| `segment-request` | `segment-meta` + N×`segment-chunk` or `segment-error` | Segment at timestamp |
| `segment-request-by-id` | `segment-meta-by-id` + N×`segment-chunk-by-id` or `segment-error-by-id` | Segment by ID |
| `segments-after-request` | `segments-after` | Up to 20 segment metas after timestamp |
| `segments-before-request` | `segments-before` | Up to 20 segment metas before timestamp |

### Segment Transfer

Segments are split into 32 KB chunks (base64-encoded JSON). The `segment-meta` message is sent first:

```json
{
  "type": "segment-meta",
  "requestTime": 1700000000,
  "segmentId": "<canonical-id>",
  "startTime": 1699999995,
  "endTime": 1700000005,
  "mimeType": "video/webm; codecs=vp8",
  "sizeBytes": 245760,
  "originMonitor": "<pubkey>",
  "contentHash": "<sha256-hex>"
}
```

Then N chunk messages:
```json
{
  "type": "segment-chunk",
  "requestTime": 1700000000,
  "startTime": 1699999995,
  "index": 0,
  "total": 8,
  "data": "<base64>"
}
```

The viewer reassembles chunks in order and verifies `contentHash` (SHA-256 of raw bytes) before resolving.

### Viewer Chains

A viewer can re-serve segments to another viewer. The `canonicalSegmentId = segment.backupOf ?? segment.segmentId` is always sent in `segment-meta` so downstream viewers set `backupOf` correctly and dedup against the original monitor's ID.

## Signal Router (`signal-router.ts`)

A single `listenForSignals` subscription handles all incoming kind-1059 events and dispatches based on `msg.type`:

- `offer-request`, `answer`, `hangup` → monitor handler (`handleOfferRequest`, `handleAnswer`, `handleHangup`)
- `offer`, `hangup`, `ping/pong` → viewer handler (`handleViewerSignal`)

The router is started by `SentrySection` when the monitor arms, and a separate viewer subscription (`ensureViewerSubscription`) is started when the viewer first connects. The module-level `seenEventIds` set in `signaling.ts` deduplicates across both subscriptions running in the same browser context.
